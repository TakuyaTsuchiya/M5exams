# Lessons Learned

## 2026-04-04: S6課題の推敲で得た教訓

### 1. 台本（シラバス）を必ず確認してからツールを選定する
- S6はGoogle Workspace Studioが正しかったのに、設計仕様書だけ見てGASで課題を作ってしまった
- 初稿の時点で台本を読んでいれば手戻りを防げた
- **台本は課題設計の一次ソース。設計仕様書やhandoffだけで判断しない**

### 2. 提出物はテキストで提出できるものに限定する
- Workspace StudioのワークフローはUI上で構築するのでそのまま提出できない
- 「プロンプト＋出力結果」のテキスト提出パターンに落とし込む必要がある
- S1〜S4と同じ「プロンプト＋出力」形式に統一すると受講生も採点者も迷わない

### 3. テンプレートの「一部だけ変えさせる」が最もシンプルな課題設計
- 全体を自由に作らせると難易度・採点コストが上がる
- テンプレートの背景欄だけ書き換えさせることで、変更範囲が明確になり採点も容易
- 「どこを変えればいいか」を視覚的に示す（オレンジ色マーカー等）

### 4. UIのボタン名は英語のまま記載する
- Workspace StudioのUI（Turn on, Save changes, Test run, Start, Run Completed）は英語
- 日本語に意訳すると受講生が画面で見つけられない
- ボタン名はそのまま英語で「」で括って記載する

### 5. 手順はダラッと文章で書かず箇条書きのステップ・バイ・ステップにする
- 「○○してから、○○して、○○してください」は読みにくい
- 番号付きリストで1アクション1項目にする
- 特に重要な注意点（Save changes忘れ、変数を消さない等）は赤字太字で目立たせる

### 6. 英語のプロンプトは日本語訳を提供する
- テンプレートのデフォルトプロンプトが英語だと受講生が読めない
- 日本語訳を課題文に載せて「コピペ→背景だけ書き換え」の流れにする
- ルーブリックにも英語全文は不要。日本語で要旨だけ書けば十分

### 7. 動作検証は複数パターンで行う
- 背景欄を変えて実際にStudioでテスト実行し、出力がちゃんと変わることを確認した
- デフォルトのまま→一般ニュース、不動産営業→不動産ニュース、エンジニア→技術ニュース
- Save changes忘れで出力が変わらないケースも発見→ルーブリックに判定ポイントとして追記

## 2026-04-04: S5課題の推敲で得た教訓

### 1. AIが出力するものは受講生に書かせない
- MermaidコードもI/O定義もAIが十分な品質で出力する
- 受講生の仕事は「AIに正しく依頼し、結果を確認すること」
- 「受講生が自力で書く」前提で課題を設計すると実態と乖離する

### 2. 提出物テンプレートは全プレースホルダーを用意する
- 「以下同様に記入」ではなく、6ステップなら6ステップ分すべての枠を用意する
- 受講生が迷わない＝採点もブレない

### 3. ルーブリックは課題の難易度に合わせる
- AIがやる課題に6観点のルーブリックは大げさ
- シンプルな課題にはシンプルなルーブリック（2観点で十分）
- 「採点のための採点基準」にしない

### 4. 比喩より直接的な表現の方がわかりやすい
- 「料理のレシピに例えると」のような比喩は回りくどくなる
- 「全体像を示す」「その具体を埋める」のように直接書く方がエレガント

### 5. 動作検証してから課題を確定する
- 実際にAIにプロンプトを投げて出力を確認する
- 想定と実際の出力レベルが違うと、課題設計全体が変わる（S5では入出力定義の手順を全面変更した）

### 6. S6への接続を意識したテーマ設定
- S5のテーマ（応募受付→自動返信）はS6でGWSで実装することを前提に選定した
- Googleフォーム→スプレッドシート→メールの流れはGASで最もシンプルに実装できる

## 2026-03-28: S3課題の推敲で得た教訓

### 1. ヒントではなく手順として指示する
- 「ヒント」ボックスに書くと曖昧になる。手順なら手順として直接指示する
- 「〜してみてください」ではなく「〜してください」

### 2. 課題文は徹底的に削ぎ落とす
- 箇条書き・補足ボックス・注釈を安易に増やさない
- 一文で伝わるなら一文で済ませる
- 「その他の設定」のような中途半端なセクションは作らない

### 3. 会社説明は文中に自然に溶け込ませる
- NG: 「株式会社ミゾリーチ（HRテック企業・従業員300名）」← 括弧補足
- OK: 「人材の転職スカウトサービスを運営する株式会社ミゾリーチ」← 文の流れ

### 4. 評価対象は「人間が書いたパート」に限定する
- カスタム指示・プロンプトの具体性を評価する
- AIの出力結果そのものは評価しない
- これを解説文に明記する

### 5. 受講生に書かせる部分を増やす＝評価の幅が広がる
- テスト入力を固定せず、プロンプト自体を受講生に設計させる
- 「職種」「知りたい項目」を自分で選ばせると楽しさも出る
- 固定テスト2回より、自由プロンプト1回の方がシンプルで評価しやすい

### 6. ラベルではなく文章で書く
- NG: 「場面設定：採用責任者への月次報告用レポートを作成します。」← ラベル付き
- OK: 「採用責任者への月次報告用レポートを作成します。」← そのまま文章

### 7. 業務の実務感を入れる
- 「毎月〜四半期ごとに繰り返す定型業務」→ なぜGPTに任せるかの説得力が出る
- 「採用責任者への報告」→ 誰のために作るかが明確になり、プロンプトの方向性が定まる
- 抽象的な「テスト実行」ではなく、具体的な業務シーンに落とし込む
