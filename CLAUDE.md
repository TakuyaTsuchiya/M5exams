# M5実習課題 プロジェクトルール

## プロジェクト概要
AIスクールのモジュール5（M5）実習課題。S1〜S6＋まとめの7セクションで構成。
受講生がChatGPT GPT、NotebookLM、Google Workspace等を使い、段階的にAI活用スキルを習得する。

## セクション構成
- S1: Botを作る（カスタム指示でフォーマット統一）
- S2: Botに知識を足す（RAG / NotebookLM）
- S3: Botに機能を足す（Web検索 / GPT）
- S4: 事例を参考に自分の業務に合わせる
- S5: 単発→ワークフローへ。業務を設計する
- S6: 設計図を実際に動かす（Google Workspace）
- まとめ: 全体統合

## ファイル構成と役割分担

各セクションには3種類のHTMLファイルがある：

| ファイル | 対象読者 | 内容 |
|---|---|---|
| `M5_S{n}_question.html` | 受講生 | 課題文（手順・提出物テンプレート） |
| `M5_S{n}_explanation.html` | 受講生 | 解説（学びのポイント・実務での活用イメージ） |
| `M5_S{n}_rubric.html` | マネージャー（採点者）のみ | ルーブリック（採点基準・目線合わせ・フィードバック例） |

その他：
- `index.html` — レビューサイトのタブUI（全ファイルをiframeで表示）
- `auth.js` — レビューサイトの認証
- `tasks/design_spec.md` — 全セクションの設計仕様書
- `tasks/lessons.md` — 推敲で得た教訓
- `drafts/` — 動作検証用のサンプル入出力

## 解説とルーブリックの分離ルール（重要）

### 解説（explanation）＝受講生向け
書くこと：
- この課題のねらい（受講生が何を学んだか）
- 技術や概念の解説（RAGとは、Web検索の活用など）
- スキルのコツ・ポイント
- 実務での活用イメージ
- AI溝口への誘導リンク

書かないこと：
- 採点基準・配点
- OK/惜しい/NGの判定基準
- フィードバックの文例
- 採点時の注意事項
- マネージャー向けの目線合わせ

### ルーブリック（rubric）＝マネージャー（採点者）向け
書くこと：
- 採点基準テーブル（観点・配点・OK/惜しい/NG）
- 採点の注意事項
- フィードバックのポイント（高評価・改善・再提出のケース別）
- 「評価対象は人間が書いたパート（カスタム指示・プロンプト等）であり、AIの出力内容は評価しない」旨の明記

書かないこと：
- 受講生向けの学習コンテンツ

## レビューサイトでの表示
- question・explanation・rubricの3ファイルすべてがレビューサイト（GitHub Pages）に表示される
- ただしrubricは受講生からは見えない仕様（LMSでは非公開）
- レビューサイトはマネージャーの目線合わせ用

## 課題文の書き方ルール

### 文体
- カッチリ寄り。「〜してください」
- 一文を短く、冗長な理屈・説教調は削る
- 「やる前に語る」より「さっさと作ろう」感を重視

### わかりやすさ
- 一文目で「何をするか」を言い切る
- 読めばわかることを書かない
- ヒントではなく手順として直接指示する
- ラベル（「場面設定：」等）ではなく文章で書く
- 会社説明は括弧補足ではなく文中に自然に溶け込ませる

### 評価方針（全課題共通）
- 評価対象はプロンプト（カスタム指示・設計文）であり、AIの出力内容そのものは評価しない
- 30点満点（全セクション共通）
- OK / 惜しい / NG の3段階

### 架空企業
- 株式会社ミゾリーチ：人材の転職スカウトサービスを運営するHRテック企業
- S2以降で共通して登場する
